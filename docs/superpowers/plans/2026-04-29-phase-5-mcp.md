# Phase 5 — MCP Server (graph queries + import/export tools)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. The MCP transport is small but precise; tool-handler implementations are mostly delegate-friendly.

**Pre-requisite:** Phase 1 complete (model). Each of Phases 2/3/4 unlocks an additional pair of MCP tools (`import_<format>`, `export_<format>`) but the graph-query subset can ship the moment Phase 1 is on `main`.

**Goal:** Provide an MCP server that lets LLMs (Claude, Qwen, etc.) interrogate the Groundwork catalogue in a structured way: list entities, walk the dependency graph, compute blast radius, generate deployment plans, and round-trip IaC.

**Architecture:** A new binary `groundwork-mcp` in the existing crate. It speaks **MCP over stdio** (JSON-RPC 2.0 framed by `Content-Length` headers, per the MCP spec). It talks to a running Groundwork HTTP server (configurable URL via `GROUNDWORK_URL`, default `http://localhost:3000`) for all reads and import/export operations. Graph traversal logic lives client-side in the MCP binary, computed against a snapshot fetched from the HTTP API.

**Tech Stack:** Rust 2021. For MCP transport, prefer the [`rmcp`](https://crates.io/crates/rmcp) crate (the Rust MCP SDK from the MCP organisation) if it is on stable crates.io and supports stdio. If `rmcp` is not suitable, fall back to a hand-rolled minimal MCP server (≈300 lines) built on `tokio::io::stdin`/`stdout` with `serde_json`. Make this decision in Slice 0; do not waste time on it.

## MCP tool catalogue

| Tool name | Inputs | Output | Notes |
|---|---|---|---|
| `catalog.list` | `entity: "deployable" \| "service" \| "exposes" \| "dependency" \| "contract" \| "sla"` | JSON array of records | Wraps `GET /<entity>/api`. |
| `catalog.get` | `entity`, `id: string` | One record or null | Wraps `GET /<entity>/api/<id>`. |
| `catalog.search` | `entity`, `query: { name?, ... }` | JSON array | Wraps `GET /<entity>/api?<filters>` (or POSTs a graphql vector query). |
| `graph.blast_radius` | `service_id: string`, `depth?: int (default 5)` | JSON tree of dependents | "If this service goes down, which deployables — and which services those deployables expose — break, transitively?" Walks `dependency` edges in reverse + `exposes` edges forward. |
| `graph.dependencies_of` | `deployable_id: string`, `depth?: int (default 5)` | JSON tree | What does this deployable consume? Walks `dependency` edges forward. |
| `graph.deployment_plan` | `deployable_id: string` | JSON list of (deployable_id, order) tuples | Topological order: dependencies first. If a dependency has no Deployable behind it (managed external service), it is listed as an external prerequisite. |
| `import.kubernetes` | `yaml: string` | ApplyReport | Wraps `POST /import/kubernetes`. |
| `export.kubernetes` | `deployable?: string`, `namespace?: string` | YAML text | Wraps `GET /export/kubernetes`. |
| `import.ansible` | `yaml: string` | ApplyReport | Wraps `POST /import/ansible`. |
| `export.ansible` | `team?: string` | YAML text | Wraps `GET /export/ansible`. |
| `import.terraform` | `hcl: string` | ApplyReport | Wraps `POST /import/terraform`. |
| `export.terraform` | `provider: "aws" \| "gcp" \| "azure"`, `deployable?: string` | HCL text | Wraps `GET /export/terraform`. |

Each tool has a JSON Schema for inputs (used in the MCP `list_tools` response) and a Markdown description that nudges the LLM toward correct usage.

### Task 1B: Implement `Snapshot`

**Files:** `groundwork/src/mcp/graph.rs`

`Snapshot` is the in-memory aggregate the MCP binary builds once per request (or caches with TTL):

```rust
pub struct Snapshot {
    pub deployables_by_id:  HashMap<String, Deployable>,
    pub services_by_id:     HashMap<String, Service>,
    pub exposes:            Vec<Exposes>,
    pub dependencies:       Vec<Dependency>,
    /// service_id → list of deployables that expose it
    pub deployables_for_service: HashMap<String, Vec<String>>,
    /// service_id → list of deployables that depend on it
    pub dependents_of_service: HashMap<String, Vec<String>>,
    /// deployable_id → list of services it depends on
    pub services_for_deployable: HashMap<String, Vec<String>>,
}

impl Snapshot {
    pub async fn build(client: &GroundworkClient) -> anyhow::Result<Self>;
    pub fn blast_radius(&self, service_id: &str, depth: usize) -> Value;
    pub fn dependencies_of(&self, deployable_id: &str, depth: usize) -> Value;
    pub fn deployment_plan(&self, deployable_id: &str) -> Value;
}
```

- [ ] **Step 1B.1: Delegate the data-loading half**

> Implement `pub async fn build(client: &GroundworkClient) -> anyhow::Result<Self>`. Fetch all entities in parallel via `tokio::join!`. Build the indices.

- [ ] **Step 1B.2: Delegate `blast_radius`**

> Walks reverse-dependency edges from `service_id`. At each step, the dependents are deployables that depend on the current service, and the recursive expansion follows their *exposed services* outward. Return a JSON object:
>
> ```json
> {
>   "service_id": "...",
>   "service_name": "payments-api",
>   "depth": 5,
>   "direct_dependents": [
>     { "deployable_id": "...", "deployable_name": "checkout", "exposes": [...] }
>   ],
>   "transitive_dependents": [...]
> }
> ```
>
> Cap depth to prevent runaway cycles. Track a visited set.

- [ ] **Step 1B.3: Delegate `dependencies_of`**

> Forward-walks dependency edges from `deployable_id`. For each dependency: if the depended-on service has a Deployable behind it (via `deployables_for_service`), recurse on that Deployable's dependencies. Return a JSON tree.

- [ ] **Step 1B.4: Implement `deployment_plan` (do this yourself; topological sort is subtle)**

Algorithm:
1. Start with `deployable_id`.
2. Traverse forward: collect every Deployable that this one depends on (via Service → exposing Deployable).
3. For each Service that has no Deployable behind it, mark it as an external prerequisite.
4. Run a Kahn-style topological sort over the collected Deployables. Detect cycles → return an error JSON object.
5. Output JSON:

```json
{
  "deployable_id": "...",
  "deployable_name": "checkout",
  "external_prerequisites": [
    { "service_id": "...", "service_name": "stripe" }
  ],
  "ordered_deployments": [
    { "order": 0, "deployable_id": "...", "deployable_name": "auth" },
    { "order": 1, "deployable_id": "...", "deployable_name": "checkout" }
  ]
}
```

- [ ] **Step 1B.5: Inline unit tests for graph functions**

Construct fake Snapshots in tests:

## Slice 2 — MCP transport + tool registration

### Task 2A: Wire the transport

**Files:** `groundwork/src/bin/groundwork-mcp.rs`, `groundwork/src/mcp/mod.rs`

The transport reads JSON-RPC requests from stdin and writes responses to stdout. Each request is one of:

- `initialize` → respond with server info.
- `list_tools` → respond with the tool catalogue.
- `call_tool { name, arguments }` → dispatch to the matching tool handler, return the result.

Implementation choice (per Slice 0):
- **If `rmcp`:** use `rmcp::server::Server::new()`, register tools via the macro/decorator API, run on stdio.
- **If hand-rolled:** read 1-line JSON-RPC frames (newline-delimited JSON; the LSP-style `Content-Length` framing is *also* supported but most stdio MCP clients use line-delimited). Dispatch with a match on `request.method`. Use `serde_json::Value` for everything.

- [ ] **Step 2A.1: Implement transport**

If hand-rolling, the skeleton:

```rust
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};
use serde_json::{json, Value};

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let base_url = std::env::var("GROUNDWORK_URL").unwrap_or_else(|_| "http://localhost:3000".into());
    let client = std::sync::Arc::new(groundwork::mcp::client::GroundworkClient::new(base_url));

    let stdin = tokio::io::stdin();
    let mut reader = BufReader::new(stdin).lines();
    let mut stdout = tokio::io::stdout();

    while let Some(line) = reader.next_line().await? {
        let req: Value = match serde_json::from_str(&line) {
            Ok(v) => v,
            Err(e) => { write_err(&mut stdout, None, -32700, &e.to_string()).await?; continue; }
        };

        let id = req.get("id").cloned();
        let method = req.get("method").and_then(|m| m.as_str()).unwrap_or("");

        let resp = match method {
            "initialize"  => initialize_response(),
            "tools/list"  => list_tools_response(),
            "tools/call"  => call_tool(&client, req.get("params").cloned().unwrap_or(Value::Null)).await,
            other         => Err(format!("unknown method: {other}")),
        };

        let frame = match resp {
            Ok(result) => json!({"jsonrpc":"2.0","id":id,"result":result}),
            Err(msg)   => json!({"jsonrpc":"2.0","id":id,"error":{"code":-32601,"message":msg}}),
        };
        let line = serde_json::to_string(&frame)?;
        stdout.write_all(line.as_bytes()).await?;
        stdout.write_all(b"\n").await?;
        stdout.flush().await?;
    }
    Ok(())
}
```

The `tools/list` and `tools/call` method names match the official MCP wire format.

```rust
use tokio::process::{Command, ChildStdin, ChildStdout};
use tokio::io::{AsyncBufReadExt, AsyncWriteExt, BufReader};

pub struct McpClient {
    stdin:  ChildStdin,
    stdout: BufReader<ChildStdout>,
    next_id: u64,
}

impl McpClient {
    pub async fn spawn(server_url: &str) -> anyhow::Result<Self> {
        let mut child = Command::new(env!("CARGO_BIN_EXE_groundwork-mcp"))
            .env("GROUNDWORK_URL", server_url)
            .stdin(std::process::Stdio::piped())
            .stdout(std::process::Stdio::piped())
            .stderr(std::process::Stdio::inherit())
            .spawn()?;
        let stdin  = child.stdin.take().unwrap();
        let stdout = BufReader::new(child.stdout.take().unwrap());
        Ok(Self { stdin, stdout, next_id: 1 })
    }

    pub async fn call(&mut self, method: &str, params: serde_json::Value) -> anyhow::Result<serde_json::Value> {
        let id = self.next_id; self.next_id += 1;
        let frame = serde_json::json!({"jsonrpc":"2.0","id":id,"method":method,"params":params});
        let line = serde_json::to_string(&frame)?;
        self.stdin.write_all(line.as_bytes()).await?;
        self.stdin.write_all(b"\n").await?;
        self.stdin.flush().await?;

        let mut buf = String::new();
        self.stdout.read_line(&mut buf).await?;
        let resp: serde_json::Value = serde_json::from_str(&buf)?;
        if let Some(err) = resp.get("error") { anyhow::bail!("MCP error: {err}"); }
        Ok(resp.get("result").cloned().unwrap_or(serde_json::Value::Null))
    }
}
```

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| `rmcp` is incomplete or not on crates.io. | The hand-rolled fallback is ~300 lines and well-spec'd; budget time accordingly. |
| MCP wire format evolves. | Pin to a spec version and document it in the README. Bump as needed. |
| Long graph traversals could hang the server. | Hard-cap depth at 10; cycles are detected via visited set. |
| Caching the snapshot per-call is expensive on a large catalogue. | First implementation: rebuild the snapshot per tool call. If profiling shows it's a bottleneck, add a 5-second TTL cache. Don't optimise prematurely. |
| The MCP binary is coupled to the HTTP API. | Acceptable for v0.2. A future v0.3 might let the MCP binary embed Groundwork's storage directly via the same `meshql-core` traits. |
